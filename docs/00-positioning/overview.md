# 定位总览

RunDoc 是一个项目变更驱动的文档自动维护系统。

## 解决什么问题

传统文档站有三个根本问题：

1. **文档滞后于代码**：代码改了，文档没人同步更新
2. **文档是死资源**：静态站点对 AI agent 不友好，无法检索、引用、更新
3. **知识散落各处**：决策在邮件、规则在 Wiki、流程在老大的脑子里

RunDoc 的核心思路是：**让项目变化成为文档更新的触发器**，而不是依赖人工"想起来去更新文档"。

### 真实场景：文档腐烂是如何发生的

**场景 A：API 变更无人同步**

```text
周一：后端把 POST /api/order 的 amount 字段改成了 total_amount
周三：前端按新字段对接，一切正常
周五：新同事看文档，按 amount 字段调用 —— 报错
      在群里问："文档写的 amount，怎么不行？"
      回答："哦对，忘了更新文档了"
```

传统系统的反应：依赖有人"想起来"更新文档。结果是永远想不起来。

RunDoc 的反应：Git diff 检测到 `order.ts` 中 `amount` → `total_amount` → 自动生成 `docs/03-technical/api-routes.md` 的补丁 → 提交 MR → 审阅通过，文档与代码同步。

**场景 B：决策信息散落**

```text
技术选型讨论：
- 架构师在 Slack 说"用 Redis 吧"
- TL 在邮件里说"Redis Cluster，注意持久化配置"
- 实际部署用的是 Redis Sentinel
- 文档里写的是"缓存方案待定"
```

传统系统的反应：信息散落在多个渠道，文档永远是最后一个被更新的。

RunDoc 的反应：通过集成 Slack、Issue、MR 描述等来源（可选），Agent 发现不一致 → 生成文档补充建议 → 人工确认后更新。

**场景 C：新人入职的知识迷宫**

```text
新人第一天："我需要了解项目的 API 设计规范"
- 打开 Wiki：找到 2022 年的版本
- 打开 Notion：找到 2023 年迁移到 v2 的记录
- 打开代码：发现实际已经是 v3 了
- 结论：不知道该信哪个
```

RunDoc 的反应：代码变更是文档更新的唯一权威触发源。代码是最新的 → 文档就是最新的。

## 与其他工具的比较

| 维度 | RunDoc | VitePress | Docusaurus | Notion | Confluence |
|------|--------|-----------|------------|--------|------------|
| **核心定位** | 文档自动维护 | 静态站点生成 | 静态站点生成 | 协作知识库 | 企业 Wiki |
| **AI 原生** | 是（设计起点） | 否 | 否 | 部分（AI 问答） | 部分（AI 搜索） |
| **更新方式** | 项目变化自动触发 | 人工编写 + 提交 | 人工编写 + 提交 | 人工编辑 | 人工编辑 |
| **变更感知** | 自动检测 Git diff | 无 | 无 | 无 | 无 |
| **文档补丁** | AI 生成 + 人工审阅 | 无 | 无 | 无 | 无 |
| **引用粒度** | 段落/块级 ID | 页面级 | 页面级 | 块级（内部） | 页面级 |
| **与代码同步** | 自动 | 手动 | 手动 | 手动 | 手动 |
| **一致性检查** | 自动交叉校验 | 无 | 无 | 无 | 无 |
| **渲染层** | 可搭配任何 SSG | 自带 | 自带 | 自带 | 自带 |
| **开源** | 是 | 是 | 是 | 否 | 否 |
| **适合场景** | 代码驱动、频繁变更 | 技术文档站点 | 技术文档站点 | 通用协作 | 企业知识管理 |

**关键差异**：VitePress / Docusaurus 解决的是"文档怎么展示"，Notion / Confluence 解决的是"文档怎么写（协作）"，RunDoc 解决的是"文档怎么保持最新"。RunDoc 可以与 VitePress 或 Docusaurus 配合使用——前者维护内容，后者渲染内容。

## 关键差异点

### 1. 双向绑定：代码 ↔ 文档

其他工具中，代码和文档是两条平行线。开发者改了代码，需要在另一条线上手动更新文档。RunDoc 建立了两者之间的自动绑定：

```text
代码变更 → RunDoc 检测 → 影响分析 → 文档补丁
```

这不是"代码即文档"（把文档写在代码注释里），而是"代码驱动文档"（代码变化触发文档更新，但文档仍是独立、人类可读的 Markdown）。

### 2. 文档作为一个可查询的知识图谱

RunDoc 的文档不是页面集合，而是一个可查询的知识网络：
- 每个段落有唯一 ID
- 文档之间有显式交叉引用
- 元数据标记了内容类型、状态、负责人
- AI 可以精确检索到段落级粒度

### 3. 运行闭环而非单次快照

传统文档是"快照"——某一时刻的静态记录。RunDoc 构建的是"闭环"：
- 持续感知变化
- 持续分析影响
- 持续生成补丁
- 持续校验一致性

文档从"静态资产"变成"持续维护的知识系统"。

## 不是什么

RunDoc 不是又一个 Markdown 静态站点生成器。它不替代 VitePress、Docusaurus 等工具。

RunDoc 的核心价值在于感知变更并驱动文档更新。当前仓库的文档站 UI 只是渲染层，真正的引擎工作在 `.rundoc/` 状态与报告系统中。

**RunDoc 不是/不做的**：
- **不是**代码注释生成器（不会把 `//` 注释变成文档）
- **不是**API 文档自动生成工具（Swagger/OpenAPI 更适合单个 API 的 reference）
- **不是**知识问答机器人（不替代 ChatGPT、Kapa.ai 等对话工具）
- **不是**项目管理工具（不管理 Issue、Task、Sprint）
- **不**自动合并文档变更（始终需要人类审阅确认）
- **不**修改代码文件（只读代码，只写文档）
- **不**替代人工判断（业务语义的正确性仍由人类确认）

## 适用场景

- 需要文档与代码保持同步的中小型技术团队
- 需要 AI agent 可读的结构化文档的系统
- 产品规则、业务流程频繁变化的项目
- 多人协作、文档审阅需求明确的团队
- 在 Git 仓库中管理文档、希望文档变更可追溯的团队
- 正在或计划使用 AI Agent 辅助开发的团队（Rundoc 提供结构化上下文）

详细判断标准：
- 你的项目代码每周有 5 次以上提交 → RunDoc 的价值随变更频率增长
- 你的团队有 3 人以上且文档消费方不只有作者本人 → RunDoc 的审阅流程有价值
- 你曾因为文档过时导致过线上问题 → RunDoc 直接解决这个痛点
- 你想让 AI Agent 理解你的项目 → RunDoc 是 AI Agent 的知识基础

## 产品边界：RunDoc vs RunMind

RunDoc 和 RunMind 是两个独立的产品，在**文档归属范围**上有明确的边界。

### RunDoc：项目级文档协作与自动捕获

- **文档归属**：文档绑定到项目
- **核心问题域**：这个项目做得怎么样了？需求是什么？技术设计是怎样的？谁改了什么？进度如何？有哪些发布和复盘？
- **典型文档类型**：需求文档、技术方案、API 设计、上线检查清单、迭代复盘、变更日志
- **受众**：项目成员（开发者、PM、QA）
- **生命周期**：跟随项目，项目归档后文档进入只读

### RunMind：组织级知识共享（不同产品）

- **知识归属**：知识属于组织，跨项目复用
- **核心问题域**：组织内的通用规则、技术决策、最佳实践是什么？AI Agent 应该遵循哪些上下文？
- **典型文档类型**：编码规范、架构决策记录、通用故障处理 SOP、团队工作流
- **受众**：全组织成员及 AI Agent
- **生命周期**：长期维护，持续演进

### 文档晋升机制

当 RunDoc 中的某篇文档具有组织级复用价值时，可以将其"晋升"到 RunMind：

```text
RunDoc 项目文档 → 人工判断具有跨项目复用价值 → 晋升到 RunMind → 组织级知识库
```

晋升意味着：
- 文档的归属从"项目"变为"组织"
- 文档脱离项目生命周期，进入组织知识库的长期维护
- 后续修改不影响原项目的文档快照

### 编辑器的共享与边界的区分

RunDoc 和 RunMind 可能共享同一个 Markdown 编辑器组件（`@runos/editor-markdown`），但**产品边界不在于编辑器，而在于业务对象**：
- RunDoc 操作的对象是"项目下的文档"
- RunMind 操作的对象是"组织下的知识条目"

判断一个功能属于哪个产品，问一句：**"这个文档是跟着项目走，还是跟着组织走？"**

## 不适用场景

- 纯内容网站（博客、营销站）—— 使用 VitePress / Docusaurus / Astro 更合适
- 不需要频繁更新的静态手册 —— 一次性编写后无需维护
- 团队规模很小、文档变更很少的项目 —— RunDoc 的收益小于维护成本
- 文档内容与代码无关的纯业务文档 —— 没有代码变更可感知
- 对文档格式有严格排版要求的场景 —— Markdown 的排版能力有限
- 团队尚未接受 AI 辅助工作流 —— 如果对 AI 生成内容接受度低，审阅负担可能超过收益
