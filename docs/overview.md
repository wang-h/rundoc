# 概览

RunDoc 是一个项目变更驱动的文档自动维护系统。

它绑定项目仓库，持续感知代码与配置变化，判断哪些文档受影响，并自动生成可审阅的 Markdown 更新。

## 核心定义

RunDoc 不以”人工整理材料”为主流程，不要求先手工产出 `meeting-notes.md` 或 `manual-notes.md` 再汇总。

RunDoc 的主流程是：

```text
项目变化 -> 影响分析 -> 文档补丁 -> 审核合并
```

### 为什么选择 AI Native 路线

传统文档工具的核心假设是”人类是唯一的文档消费者和生产者”。但在 AI Agent 日益参与软件开发的今天，这个假设已经过时。

**文档系统的两类消费者正在合流**：

1. **人类开发者**：需要准确的、结构化的知识来理解系统
2. **AI Agent**：需要可检索、可引用、可校验的知识来辅助开发

传统文档对 AI 不友好：
- 页面结构靠视觉排版，AI 难以解析
- 没有稳定的引用 ID，AI 的回答无法溯源
- 更新靠人工触发，信息随时间衰减

RunDoc 选择 AI Native 路线，意味着：
- 每个文档段落都被设计为 AI 可寻址的知识单元
- 文档更新由项目变化自动触发，而非等待人工发起
- AI 不仅消费文档，还参与文档的维护——作为”起草者”而非”决策者”

这是一种根本性的架构选择：**文档从”给人看的死文本”变成”人机共用的活知识”**。

## 输入与输出

输入（项目状态）：

- Git 差异
- 提交历史
- GitLab Issue / MR（可选）
- 现有 `docs/`
- `.rundoc/config.yml`
- 可选人工材料（只作为补充）

输出（文档变更资产）：

- `docs/**/*.md` 更新
- `docs/04-ai/*` AI 上下文更新
- 文档一致性检查结果
- `.rundoc/reports/*.md` 扫描报告
- 草稿提交 / MR 建议

### 输入输出示例

**示例：一次典型的 API 变更**

```text
输入：
  Git diff 显示 src/api/order.ts 中：
    - export interface CreateOrderParams {
    -   amount: number       // 删除了
    +   total_amount: number  // 新增
    +   currency: string      // 新增

输出：
  docs/03-technical/api-routes.md 的补丁：
    - | `amount`    | number | 必填 | 订单金额 |
    + | `total_amount` | number | 必填 | 订单总金额 |
    + | `currency`      | string | 必填 | 货币代码（ISO 4217） |
  
  .rundoc/reports/scan-2024-03-15.md：
    变更来源: src/api/order.ts (commit a1b2c3d)
    影响文档: docs/03-technical/api-routes.md (行 42-58)
    变更类型: API 参数变更
    建议: 同步更新相关的错误码章节
```

## 运行闭环

```text
Detect -> Understand -> Map -> Patch -> Commit
```

### 第一步：Detect（发现变化）

**做什么**：感知项目状态中发生的任何变化。

- 监听 Git 提交（新增、修改、删除的文件及具体 diff）
- 可选监听 GitLab Issue / MR 的状态变化
- 可选感知 `.rundoc/config.yml` 配置的变更
- 对比上一次扫描快照，生成变化清单

**产出**：变化清单 —— 哪些文件变了、谁变的、什么时候变的、变了什么

### 第二步：Understand（理解影响）

**做什么**：分析每项变化的语义含义，判断它对文档体系的影响。

- 解析代码变更的语义（新增 API？修改类型？废弃参数？配置变更？）
- 判断变更类型：breaking change / non-breaking change / addition / deprecation
- 识别受影响的领域：产品规则、API 规范、架构设计、运维流程
- 标记变更的重要程度：critical / major / minor

**产出**：影响分析清单 —— 每项变化的类型、领域归属、严重程度

### 第三步：Map（映射到文档）

**做什么**：将每项变化映射到具体受影响的文档和段落。

- 通过代码路径推断相关文档（`src/api/order.ts` → `docs/03-technical/api-routes.md`）
- 通过元数据交叉引用定位（文档中引用了变更涉及的模块）
- 通过内容语义匹配发现隐含关联（文档描述的概念与变更相关）
- 输出"需要更新"的精确位置：文件路径 + 行号范围

**产出**：文档影响地图 —— 哪些文档的哪些段落需要更新，以及原因

### 第四步：Patch（生成补丁）

**做什么**：为每个受影响的文档段落生成具体的 Markdown 补丁。

- 生成标准的 Git diff 格式补丁
- 补丁范围精确到段落，而非整页重写
- 附带到行级的变更说明（这个修改对应的代码来源）
- 优先更新已有内容，只在必要时新增章节
- 标记补丁的置信度：high / medium / low（影响人类审阅优先级）

**产出**：Markdown 补丁集 —— 可直接应用于 `docs/` 的 diff

### 第五步：Commit（提交审阅）

**做什么**：将补丁打包成人类可审阅的提交或合并请求。

- 生成符合项目规范的 Git commit
- 在 MR 描述中附带变更说明：
  - 触发来源（哪次提交触发了这次更新）
  - 影响范围（哪些文档被修改）
  - 每条变更的代码引用路径
- 提交到草稿分支或直接创建 MR
- 等待人类审阅后合并

**产出**：可审阅的 Git commit / MR —— 人类确认后合并入主分支

### 闭环的意义

五个步骤形成完整闭环：

```text
                            人类审阅
                               ↓
Detect → Understand → Map → Patch → Commit
   ↑                                   ↓
   └────── 合并后的文档成为新的检测基线 ←┘
```

每完成一次闭环，文档就与代码同步一次。闭环的频率取决于项目变更的频率——变更越多，闭环运转越频繁，文档与代码的差距越小。

## 与传统文档系统对比

| 维度 | 传统文档工具 | RunDoc |
|------|------------|--------|
| **内容维护模式** | 人工编写 → 人工发布 | 项目变化 → 自动检测 → AI 起草 → 人工审阅 |
| **更新触发** | 人工"想起来" | Git 提交 / 配置变更自动触发 |
| **更新粒度** | 整页重写或无人更新 | 段落级精确补丁 |
| **代码同步** | 手动（或不同步） | 自动感知，持续同步 |
| **AI 角色** | 无，或事后添加的聊天窗口 | 内建的一等使用者 |
| **一致性保证** | 人工检查（或从不检查） | 自动交叉校验 + 报告 |
| **知识腐烂** | 随时间必然衰减 | 持续维护，衰减可控 |
| **工作流集成** | 独立编辑，与开发流程无关 | 嵌入 Git 工作流，通过 MR 审阅 |

**核心差异总结**：传统文档是"存储知识的仓库"，RunDoc 是"维护知识的系统"。仓库会被动地接受过时，系统会主动地对抗过时。

## 项目驱动的文档管理

除了 AI 自动感知变更，RunDoc 还提供了以**项目（Project）**为单元的文档组织能力：

### 项目数据模型

每个项目包含：
- `id` / `slug_id`：唯一标识与 URL 友好名称
- `name` / `description`：项目名称与描述
- `git_repo_url`：关联的 Git 仓库地址
- `status`：项目状态（`active` / `archived`）

项目是文档的容器——一个项目可以包含多篇文档，文档之间支持树形嵌套（通过 `parent_doc_id` 建立父子关系）。

### 文档 CRUD API

完整的 REST API 支持对文档的全生命周期管理：

```text
POST   /api/projects/:projectId/docs    创建文档
GET    /api/projects/:projectId/docs    列出项目文档
GET    /api/projects/:projectId/docs/:id 获取单篇文档（含版本历史）
PUT    /api/projects/:projectId/docs/:id 更新文档（自动保存快照）
DELETE /api/projects/:projectId/docs/:id 删除文档（软删除）
```

每次更新文档时，系统自动保存一个**历史快照**（包含标题、内容、修改者、变更摘要），形成完整的版本记录，支持回溯查看。

### 文档编辑体验

RunDoc 集成了共享编辑器包 `@runos/editor-markdown`，基于 TipTap 3 构建，提供：
- **工具栏**：常用 Markdown 格式快捷操作（粗体、标题、列表、代码块、链接等）
- **分屏预览**：编辑与渲染预览同屏显示，所见即所得
- **自动保存**：编辑内容实时保存，每次更新自动生成版本快照

## 双重定位：AI 驱动 + 人工编辑

RunDoc 并非"全自动"或"全手动"——它同时支持两种工作模式：

| 模式 | 触发方式 | 适用场景 |
|------|---------|---------|
| **AI 自动** | Git 提交 / 配置变更自动触发 Detect→Commit 闭环 | 代码变更导致的技术文档同步 |
| **人工编辑** | 通过文档编辑器直接编写和修改 | 产品需求、设计决策、会议记录等需要人类判断的内容 |

两种模式共享同一套文档存储和版本管理体系，互不冲突。AI 生成的补丁通过 PR 审阅后合并，人工编辑的内容也可以被 AI 感知和分析。

## 数据存储

RunDoc 使用 **SQLite** 作为后端数据库，通过 **Kysely**（类型安全的 SQL 查询构建器）和 **better-sqlite3**（高性能 Node.js SQLite 驱动）进行数据访问：

- **数据库文件**：默认路径为项目根目录的 `rundoc.db`，可通过 `DATABASE_PATH` 环境变量自定义
- **WAL 模式**：启用 Write-Ahead Logging，支持并发读写
- **外键约束**：确保数据引用完整性
- **自动迁移**：服务启动时自动执行数据库迁移，无需手动初始化

前端在无法连接后端时自动降级为 **Mock 模式**，使用内置的示例数据，确保前端开发和应用体验不依赖后端运行状态。
